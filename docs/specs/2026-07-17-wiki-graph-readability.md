# Wiki Graph 可读性优化设计

## 背景

当前 Wiki Graph 在节点与引用增长后难以辨认页面名称。真实数据中，Dashboard 图在约 26 个节点、204 条原始引用时已经出现节点聚集、标签缩小并彼此覆盖；另一个 Subject 的 40 个节点对应 715 条原始引用。同一页面正文多次引用同一目标时，API 会生成多条完全重叠的边，既放大布局弹簧力，也制造不必要的视觉噪声。

现有布局还有两个放大问题：

- `cose` 初排与自定义力导模拟的理想边长仅为 90，节点斥力为 6000，密集图会压成紧凑团簇。
- 全屏模式沿用紧凑视图的节点位置，并将原缩放固定乘 2；它没有利用更大的全屏画布重新计算可容纳全图的缩放，因此标签仍偏小。

## 目标

- 同一视图中节点间距明显增大，页面名称更容易逐项扫描。
- 重复引用不再绘制为重叠边，但相反方向的引用仍分别保留。
- Dashboard 主图获得足够的稳定高度；全屏模式充分使用可用画布。
- 保留当前节点拖拽、滚轮缩放、页面聚焦、点击跳转和全屏切换时位置不重排的交互合同。

## 非目标

- 不更换 Cytoscape，也不引入新的布局插件。
- 不改变 Wiki 原始引用数据、Dashboard 顶部 Links 统计或 lint/backlink 语义。
- 不增加布局设置面板、密度滑杆或用户持久化偏好。
- 不对不同 Subject 保存节点坐标。

## 方案比较

### 方案 A：只增大 `cose` 间距参数

改动最小，但 `fit: true` 会把扩大的图重新缩回既有容器；重复边和全屏缩放问题仍在，标签可读性改善有限。

### 方案 B：聚合视觉关系 + 统一放宽布局 + 改善视口（推荐）

将同方向的重复页面对聚合为一条关系，按不同来源页面数计算节点热度；同时提高初排和持续模拟的边长/斥力，扩大 Dashboard 图高度，并在全屏容器中按实际尺寸重新 fit。该方案直接处理数据、布局、容器三个拥挤来源，且不改变底层知识数据。

### 方案 C：接入 `fcose` 等新布局插件

可能获得更强的防重叠能力，但会增加依赖、调参面与运行成本。当前问题可由已有 Cytoscape 能力解决，不符合 YAGNI。

## 设计

### 图数据投影

`GET /api/graph` 继续读取全部 Wiki 引用，但输出边时按 `sourceSlug -> targetSlug` 聚合：

- 同方向重复引用输出一条边，并携带 `weight` 表示原始引用次数。
- `A -> B` 与 `B -> A` 是两条不同关系，分别保留。
- 跨 Subject、缺失源节点或缺失目标节点的引用继续过滤。
- 节点 `linkCount` 改为指向该节点的不同来源页面数，使节点尺寸表达图拓扑中心性，而不是同一页面重复提及次数。
- `meta.referenceCount` 保留过滤后的原始引用总量，`meta.edgeCount` 表示可视化关系数。

### 布局与标签

- 理想边长由 90 提高到 140。
- 节点斥力由 6000 提高到 12000，重力由 0.3 降到 0.2。
- Dashboard 图使用移动端 28rem、桌面 36rem 的稳定高度。
- 标签字号由 10px 提高到 11px，默认不透明度由 0.6 提高到 0.78，最大宽度由 180 提高到 220；保留画布色描边，避免边线穿过文字影响识别。

### 全屏视口

全屏切换仍迁移同一个 Cytoscape 实例并冻结持续模拟，节点模型坐标不变。挂载到全屏容器并 `resize()` 后，通过 `cy.animate({ fit: { eles, padding: 64 } })` 按真实画布计算 zoom/pan 并进行 240ms 过渡；退出时恢复进入前的紧凑视口快照。

## 成功标准

- 聚合逻辑测试证明重复同向引用只生成一条边，反向引用保留，节点热度按唯一来源计数。
- Graph 布局测试锁定更宽松的最小参数，防止回归为紧凑配置。
- 真实 Subject 在 Dashboard 1440x1000 视口中，主图高度和标签清晰度较基线明显提升。
- 全屏模式使用大画布重新 fit，节点位置在切换前后不重排；退出后紧凑视口恢复。
- 窄屏无横向页面溢出，图仍可缩放、拖拽和进入全屏。
