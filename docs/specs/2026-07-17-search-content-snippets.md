# 搜索正文片段设计

## 背景

全局搜索已经通过 FTS5 同时索引页面标题、摘要和正文，搜索接口也返回 `snippet` 字段，但当前查询固定从 FTS 第 0 列（标题）生成片段。命令面板因此只能稳定展示页面标题，无法告诉用户正文中的具体命中位置；已有 `<mark>` 标记还会被 React 当作普通文本显示。

## 目标

- 正文命中时展示包含查询词的正文上下文，而不只显示页面名称。
- 命中词在片段中有清晰但克制的高亮。
- 每条结果最多展示两行片段，保留命令面板的紧凑扫描效率。
- 不使用 `dangerouslySetInnerHTML`，服务端返回内容即使含 HTML 也只能作为文本显示。
- 保持现有 Subject 隔离、排序、键盘选择和页面跳转行为不变。

## 非目标

- 不改变 FTS 分词、排序权重或检索召回参数。
- 不新增搜索结果页、筛选器或跨 Subject 搜索。
- 不把 Markdown 渲染为富文本，也不在片段中展示图片、链接或代码样式。
- 不改变 Ask AI、语义检索和 Agent 工具使用的搜索结果。

## 现状与根因

`pages_fts` 的列顺序为 `title, summary, body, subject_id, slug`。`searchPages()` 当前调用 `snippet(pages_fts, 0, '<mark>', '</mark>', '...', 32)`，其中列索引 `0` 指向标题而非正文。命令面板已有片段占位，但直接渲染字符串，因此标记不会成为高亮节点。

## 方案比较

### 方案 A：只把 snippet 列改为正文

优点：后端改动最小。缺点：前端仍会显示字面量 `<mark>` 标签，命中词不可读，也没有安全的高亮语义。

### 方案 B：正文片段配合 `dangerouslySetInnerHTML`

优点：可以直接利用 SQLite 返回的标记。缺点：把数据库文本送入 HTML 注入通道，必须额外建立完整清洗边界，对仅有文本高亮的需求过重。

### 方案 C：正文片段配合受控标记解析（推荐）

后端显式从正文列生成片段；客户端只识别固定的 `<mark>` / `</mark>` 定界符，将其拆成普通文本与命中文本，再交给 React 渲染。

优点：直接解决根因，保留高亮且没有 HTML 注入面；解析逻辑可作为纯函数单测。代价：增加一个很小的共享片段解析函数。

## 推荐设计

### 后端片段契约

- `searchPages()` 使用 FTS 正文列索引 `2` 生成片段。
- 继续使用 SQLite `snippet()` 的 32 token 上下文和 `...` 省略符。
- 使用固定 `<mark>` 定界符作为传输格式；API 继续原样返回 `{ page, snippet, rank }`，无需新增字段。
- 标题或摘要命中但正文没有同词时，正文片段可能没有高亮；这类结果仍保留页面标题，片段仅在非空时展示。

### 客户端安全高亮

- 新增纯函数把 snippet 拆为 `{ text, highlighted }[]`。
- 只识别精确、小写的受控标记；其他任意 HTML 字符串保持普通文本。
- 未闭合标记降级为普通文本，不猜测或修复 HTML。
- 命中段使用现有强调色的低对比背景和前景色，不引入新的主题颜色。

### 命令面板布局

- 页面标题保持单行截断。
- 正文片段从一行扩为最多两行，使用更宽松行高，便于理解上下文。
- 图标和跳转箭头保持固定尺寸，不因片段长度产生布局位移。
- 结果项仍是单一 `Command.Item`，不增加嵌套卡片或额外按钮。

## 成功标准

- 搜索只存在于正文的词时，结果 snippet 包含该词及前后正文，而不是页面标题。
- 命中词以 React `<mark>` 节点高亮，原始标记字符不显示。
- snippet 中的任意非受控 HTML 都按文本渲染，不执行或注入 DOM。
- 长片段最多占两行，桌面和移动宽度下标题、图标、箭头不重叠。
- 既有 Subject 隔离、FTS 更新/删除一致性测试继续通过。
