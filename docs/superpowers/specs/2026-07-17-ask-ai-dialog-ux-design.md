# Ask AI 弹窗尺寸、功能区与渲染性能优化设计

## 背景

Ask AI 已是可拖动的召唤式悬浮工作面，但其内部仍按固定宽度侧栏组织：桌面尺寸固定为 440 × 680px，会话、新建、清空、保存分散在三处；流式回答的每个 SSE delta 都同步重建最后一条 Markdown，消息列表也在每次更新时执行平滑滚动。结果是长回答中拖动滚动条会与自动滚动竞争，GFM 表格在频繁重排时出现闪烁。

## 目标

- 桌面端可分别调整宽度、高度或同时调整两者，尺寸始终留在视口安全区内。
- 会话选择、New conversation、Clear、Save to Wiki 进入同一稳定工具区。
- 用户主动向上滚动或拖动滚动条后，流式更新不再抢回底部；回到底部后恢复自动跟随。
- 合并同一渲染周期中的流式 delta，减少 Markdown 解析、React 更新与表格布局次数。
- 已完成消息不因最后一条流式回答更新而重复渲染。
- 保留移动端 Bottom Sheet、选区引用、会话恢复、审批卡片和保存任务追踪。

## 非目标

- 不引入消息虚拟列表；现有会话规模还不足以证明这项复杂度合理。
- 不把 Markdown 解析迁移到 Web Worker，也不更换现有 unified/remark 管线。
- 不改变 Ask AI 服务端问答、会话存储或 PendingAction 契约。
- 不在移动端加入 resize 手势；移动端继续使用高 88svh 的 Bottom Sheet。

## 现状与根因

1. `ask-ai-floating-panel.tsx` 用固定 Tailwind 宽高，只有位置拖动，没有尺寸状态或 resize 命中区。
2. ConversationSwitcher 独占一行并自带 New；Clear 位于 ChatInterface 的临时头部；Save 位于消息区和输入区之间。动作出现/消失会改变内容区高度，也增加扫描成本。
3. `answer-delta` 每到一个事件就调用 `setMessages`，最后一条回答会同步执行 `renderMarkdown()`。
4. MessageList 以整个 `messages` 数组为 effect 依赖，每次 delta 都调用带 `smooth` 的 `scrollIntoView()`；用户拖动浮层滚动条时仍会持续写滚动位置。
5. 消息行使用数组下标 key，且所有行由 MessageList 重新构造；虽然 MarkdownText 有 memo，行级活动、引用和外围节点仍参与协调。
6. SaveToWikiButton 的局部成功状态没有按回答变化重置；保存过一条回答后继续对话，可能仍显示上一条回答的保存状态。

## 方案比较

### 方案 A：CSS `resize: both` + 重新排列按钮

优点：实现最少，浏览器原生 resize 成本低。

缺点：只能从右下角调整，难以统一最小/最大尺寸与视口安全边界；无法提供明确命中区和键盘语义；不解决流式滚动竞争和表格重排。

### 方案 B：受控 resize + 统一工具区 + 渲染批处理（推荐）

在悬浮容器本地保存尺寸，提供右边、下边、右下角三个 resize 命中区；纯函数负责尺寸约束。ChatInterface 持有统一工具区和动作语义。流式内容按动画帧合并，消息列表用“是否贴近底部”的显式状态决定自动跟随，并把消息行 memo 化。

优点：直接覆盖三类问题；边界可测试；不扩大服务端与数据库改动；与现有拖动和移动端模型兼容。

缺点：需要谨慎协调拖动、resize、窗口变化与工具区状态。

### 方案 C：消息虚拟化 + Web Worker Markdown

优点：能支持极长会话和更重的 Markdown。

缺点：会引入动态高度测量、流式尾消息占位、交互元素跨 worker 序列化等新问题；当前性能瓶颈主要来自无节制更新和滚动竞争，违反 YAGNI。

## 推荐设计

采用方案 B。

## 视觉主张

Ask AI 是紧凑但可扩展的阅读协作工作面：平静表面、稳定工具条、单一强调色；尺寸变化只扩大工作空间，不增加装饰或层级。

## 内容计划

1. 标题栏：Ask AI、当前上下文、拖动区域、关闭按钮。
2. 功能区：当前会话选择器；New、Clear、Save 三个图标动作，稳定占位并用 tooltip/aria-label 解释。
3. 内容区：消息、审批动作；只允许消息区承担主要滚动。
4. 输入区：引用、输入、段落选择、发送/停止。

## 交互主张

- 拖动标题栏改变位置；拖动右边/下边/右下角直接改变尺寸，指针与面板边界保持跟手。
- resize 期间关闭过渡并设置全局 cursor/user-select；释放后保留本次应用会话内尺寸。
- 新消息发送时跳到底部；流式过程中仅当用户仍贴近底部时自动跟随。用户向上滚动后保持阅读位置，回到底部后自动恢复。

## 尺寸与边界

- 默认：440px 宽、`min(680px, viewportHeight - 64px)` 高。
- 最小：360 × 420px，保证会话标题、三个图标动作和输入区仍可用。
- 最大：面板当前左上角到视口右/下 16px 安全边距之间的可用尺寸。
- 窗口变小时，先把尺寸压入可用视口，再约束位置；关闭再打开保留组件存活期间的最后尺寸。
- resize 命中区不参与标题栏拖动；移动端不渲染 resize 命中区。

## 功能区

- ConversationSwitcher 只负责当前会话选择和历史会话菜单，不再自带 New。
- ChatInterface 统一提供：
  - New conversation：停止当前流，清空临时消息、审批卡、引用与确认状态，并切到未持久化新会话。
  - Clear：只清空当前可见消息和确认状态，保留当前会话 ID，延续既有语义；生成中先停止。
  - Save to Wiki：始终稳定占位；没有可保存回答或正在生成时禁用；打开锚定于按钮的标题输入浮层。
- 三个动作使用 icon-only + tooltip，避免在最小宽度下挤压当前会话标题。

## 渲染与滚动

### 流式批处理

- `answer-delta` 仍立即追加到内存字符串，但同一动画帧只提交一次 React 状态。
- `error`、流结束、停止和卸载前显式 flush/cancel，保证最后内容不丢失且不会在卸载后写状态。
- tool activity、citations、pending action 继续独立更新，不改变事件顺序语义。

### 消息渲染

- 抽出 memo 化 MessageRow；历史消息对象未变化时跳过整行协调。
- 最后一条 Assistant 内容变化时只重算该条 Markdown。
- 表格使用固定列布局、单元格换行和消息宽度约束，避免窄面板中列宽随 token 反复变化或撑宽容器；代码等不可换行内容仍由消息表面的横向滚动承接。

### 自动跟随

- 用 `scrollHeight - scrollTop - clientHeight <= 48px` 判断是否贴近底部。
- 新发送用户消息时强制跳到底部；流式更新只在贴底状态下用即时 `scrollTop` 跟随，不使用 smooth scrolling。
- 用户滚离底部后不再写 scrollTop；返回阈值内重新启用。

## 错误与边界

- resize 与标题栏拖动互斥；动作按钮、输入与 resize handle 不触发面板位置拖动。
- 当前位置太靠右/下时，允许缩小但不允许向视口外扩大；用户可先移动面板再扩大。
- New conversation 在流式中会中止 reader，防止旧回答写入新会话。
- Save 浮层在新回答、切换/新建会话后重置旧成功/错误状态。
- reduced-motion 用户不依赖动画理解 resize 或滚动状态。

## 成功标准

- 桌面可拖动右边、下边和右下角调整尺寸，最小/最大边界与窗口变化均不使关闭按钮或输入区离开视口。
- 会话选择、New、Clear、Save 位于同一功能区；动作在无内容/生成中呈现正确禁用态，布局不跳动。
- 长回答流式输出时，用户拖动滚动条离开底部后位置不被抢回；回到底部后继续跟随。
- 一批同步 delta 只触发一次内容提交；完成消息不因流式尾消息更新而重复渲染。
- 窄宽度下表格固定列宽并换行，不撑破面板；流式表格不再出现可感知的高频闪烁。
- 移动端 Bottom Sheet、选区引用、审批、历史会话和 Save to Wiki 任务不回归。
