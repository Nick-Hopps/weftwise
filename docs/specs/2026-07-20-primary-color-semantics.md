# 全站主色语义调整设计

## 背景

weftwise 当前把品牌的纬线朱定义为全局 `accent`：亮色主档为 `#CC3F27`，暗色主档为 `#FF6B4D`。所有消费 `accent` 的主按钮、开关、焦点环、当前导航、进度状态与选中反馈因此都呈现为红橙色。

危险与失败状态虽然使用独立的 `danger` 绯红，但两组颜色在色相和视觉联想上仍然接近。结果是正常操作也容易被理解为警告、失败或破坏性操作，降低操作反馈的可辨识度。

## 目标

- 正常操作、当前状态与焦点反馈不再以红色为主。
- 危险、失败和校验错误继续独占红色语义。
- 保留 weftwise 现有的经线靛与纬线朱品牌资产，不引入第三套品牌主色。
- 通过语义 token 一次性调整现有组件，避免逐组件替换颜色或复制样式。
- 亮色与暗色主题下，主按钮文字、正文链接和危险文字保持 WCAG AA 对比度。

## 现状与约束

- `src/app/globals.css` 是颜色 token 的单一真实源，Tailwind 在 `tailwind.config.ts` 中只做语义映射。
- 组件已普遍使用 `accent`、`link`、`danger` 等语义类，说明问题位于 token 分工而不是组件调用方式。
- `--brand-warp` 与 `--brand-weft` 由标志、favicon 和品牌资产消费，不能因 UI 主色调整而改写。
- 图谱节点、正文 wikilink、Mermaid secondary 已使用经线靛；调整后需要避免结构色和操作色失去层级。
- 本轮不重构组件、不改变交互、不调整中性底色或排版。

## 方案比较

### 方案 A：经线靛承担全局操作与连接语义（采用）

将 `accent`、输入焦点、焦点环、选区和图谱激活态切换到经线靛；`link` 继续使用同一家族，但通过色阶与交互形态区分。纬线朱只保留在品牌标志和品牌资产中。

优点是直接复用现有品牌色，改动集中、风险低，操作与危险状态的色相区分最清晰。代价是“经线＝结构、纬线＝动作”的原品牌解释需要更新为“经线靛＝产品交互、纬线朱＝品牌识别”。

### 方案 B：中性黑灰作为主操作色

主按钮与选中态改为近黑，链接保留经线靛，纬线朱仅用于品牌。

优点是克制、编辑器气质强，正常操作和错误状态区分最大；缺点是主操作与普通文本、深色界面的层级较弱，品牌识别主要只剩标志。

### 方案 C：引入新的蓝绿操作色

新增青蓝或蓝绿家族专门承担 `accent`，经线靛继续负责结构，纬线朱保留品牌。

优点是三个语义角色最独立；缺点是扩张色板、稀释现有双色品牌，并增加后续维护与配色校准成本，不符合 YAGNI。

## 设计

### 色彩职责

| 职责 | 色彩家族 | 亮色主题 | 暗色主题 | 典型界面 |
|------|----------|----------|----------|----------|
| 产品操作与焦点 | warp 经线靛 | `500/600/700` | `400/200/500` | 主按钮、开关、选中态、焦点环、进度 |
| 结构与连接 | warp 经线靛 | `500/700` | `400/200` | wikilink、图谱节点、Mermaid secondary |
| 品牌识别 | weft 纬线朱 | 原品牌 token 不变 | 原品牌 token 不变 | 标志、favicon、OG 品牌资产 |
| 危险与失败 | danger 绯红 | 现有 danger token | 现有 danger token | 删除、失败、错误、校验告警 |

操作与链接虽然共享经线靛家族，但仍通过形态区分：操作使用实心背景、浅色选中面或焦点环；链接使用文字、下划线和 hover 色阶。两者不会依赖颜色单独传达用途。

### Token 调整

- `--color-border-accent`：weft 500 → warp 500。
- `--color-accent-primary/hover/active/subtle/strong-fg`：全部改由 warp 色阶组成。
- `--color-input-border-focus`、`--color-focus-ring`、`--color-selection-bg`：改用 warp。
- `--color-graph-active`：改用 warp 的强调色阶，避免正常节点交互再次呈红。
- 暗色主题补充对应的 accent、focus、selection 覆盖，主按钮保持高对比前景。
- `--brand-weft`、`--base-weft-*` 与 danger token 保留；前者继续服务品牌资产，后者作为品牌色阶储备，不再承担默认操作语义。

### 视觉层级

- 亮色主按钮采用 warp 500 配白字，hover/active 逐级加深。
- 暗色主按钮采用 warp 400 配深墨前景，hover 使用更亮的 warp 200，active 回落 warp 500。
- 浅色选中面采用 warp 50；暗色选中面采用低明度蓝灰底，不使用红棕底。
- danger 组件不改色，确保改动后错误与破坏性动作成为界面中唯一稳定的红色反馈。

## 测试与验证

1. 先增加主题语义契约测试，确认当前 `accent` 仍指向 weft 并以预期原因失败。
2. 修改 token 后运行契约测试、类型检查、全量测试、lint 与生产构建。
3. 启动真实应用，在亮色与暗色主题检查 Dashboard、Sidebar、Settings、Wiki 阅读页和任务面板。
4. 核对主按钮、当前导航、开关、输入焦点、正常进度均为靛色；删除、失败与错误仍为红色。
5. 用脚本计算关键前景/背景组合对比度，确保达到 WCAG AA。

## 风险与缓解

- **界面大面积变蓝导致品牌感减弱**：保留标志纬线朱，纸色与墨色不动；颜色变化只影响交互语义。
- **链接与操作颜色过于接近**：依赖现有按钮形态、选中面、下划线与 hover 行为区分，不增加新的色族。
- **暗色主题对比不足**：分别定义暗色 accent 前景与浅色选中面，并用数值脚本验证。
- **遗漏硬编码红色**：扫描 `red-*` 和十六进制红色；只保留真实错误、删除和 diff 删除行。

## 非目标

- 修改 logo、favicon、OG 图或品牌源 SVG。
- 调整危险色、成功色、警告色、中性色与页面布局。
- 逐组件重新设计按钮、导航、任务面板或图谱。
- 增加用户自选主题或多套配色方案。
